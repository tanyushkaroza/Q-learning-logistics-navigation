Данная работа посвящена исследованию эффективности алгоритма Q-learning для задачи поиска оптимального пути в дискретных средах. Проект включает:

*Обучение агента на множестве случайно генерируемых лабиринтов (5×5, 7×7, 10×10)
*Сравнение с эталонным алгоритмом BFS (поиск в ширину) в статической среде
*Тестирование в динамической среде с движущимися препятствиями
*Реализацию механизма локального зрения (Local Vision) для безопасной навигации
*Визуализацию траекторий в статике и GIF-анимацию движения


Архитектура проекта

- Config.py — Гиперпараметры MDP (γ, ε, размеры сетки, награды)

- MatricesForTrain.py — Генератор обучающих лабиринтов (300 шт., плотность 35%)

- MatricesForTest.py — Генератор тестовых лабиринтов (50 шт., плотность 15%)

- Environment.py — Статическая среда: step(), reset(), визуализация

- AgentForTrain.py — Q-learning агент: ε-greedy, обновление по Беллману

- AgentForTest.py — Инференс-агент: маскирование опасных действий

- Train.py — Цикл обучения с линейным затуханием ε

- Test.py — Тестирование с жадной стратегией (ε=0)

- BFS.py — Эталонный алгоритм для сравнения

- DynamicEnvironment.py — Динамическая среда + локальное зрение + анимация

- main.py — Интерактивное меню (8 режимов работы)

1)Техническое совершенствование алгоритма
*Переход к Deep Q-Networks (DQN). 
Замена табличной Q-функции на нейронную сеть позволит работать с большими и непрерывными пространствами состояний. 
Это открывает возможность навигации на картах размером 100×100 и более, где табличный метод становится невозможным из-за экспоненциального роста числа состояний.

*Использование двойного DQN (DDQN). Разделение выбора действия и оценки Q-значения помогает устранить проблему завышения оценок, 
характерную для классического DQN. Это повышает стабильность обучения на 15–20% и ускоряет сходимость.

*Приоритетный буфер воспроизведения. Вместо равномерной выборки переходов из буфера памяти, предложенный подход выбирает наиболее "полезные" переходы с высокой 
TD-ошибкой для частого повторения. Это позволяет ускорить сходимость в 2–3 раза без потери качества конечной политики.

*Алгоритмы Actor-Critic (A2C, PPO). Прямая оптимизация стратегии без хранения Q-таблицы делает эти методы устойчивыми в средах с частичной наблюдаемостью, 
где агенту недоступна полная карта препятствий.

2)Усложнение моделей среды
*Стохастическое движение препятствий. 
В текущей реализации препятствия движутся детерминированно, отскакивая от границ. 
Добавление случайности — например, изменение направления с вероятностью 0.3 на каждом шаге — сделает поведение среды более реалистичным, 
имитируя движение людей, транспорта или других непредсказуемых объектов.

*Появляющиеся и исчезающие препятствия. 
Некоторые клетки могут блокироваться или разблокироваться по таймеру или в ответ на действия агента. 
Это симулирует открывающиеся двери, разводные мосты, временные ограждения или зоны проведения ремонтных работ.

*Многоагентное взаимодействие. 
Запуск нескольких RL-агентов одновременно, которые должны избегать столкновений друг с другом, 
координировать действия и, возможно, обмениваться информацией. 
Это критически важно для роя дронов, флота AGV на складе или группы сервисных роботов в общественном пространстве.

*Непрерывное пространство действий. 
Вместо четырёх дискретных направлений агент может перемещаться с произвольным вектором скорости и направления. 
Это приближает модель к управлению реальными роботами с голономным или дифференциальным приводом.

3)Инженерные и пользовательские улучшения
*Распределённое обучение. 
Параллельный запуск нескольких экземпляров агента на GPU или 
кластере позволит ускорить эксперименты в десятки раз, что особенно важно при обучении глубоких нейронных сетей.

*Веб-интерфейс. 
Создание интерактивной карты, куда пользователь может загрузить свой лабиринт, 
вручную расставить препятствия и наблюдать за движением агента в реальном времени. Это сделает технологию доступной для не-программистов.

*API для интеграции. 
Разработка REST API позволит вызывать обученного агента из внешних систем — ERP, WMS (систем управления складом), ROS (Robot Operating System). 
Это критически важно для внедрения в реальные бизнес-процессы.
